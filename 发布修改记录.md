# 发布修改记录

## Go2 多速度与斜坡课程 Demo

### 背景

早期路线是从 `go2_v2` 继续训练多速度和斜坡课程，但后续发现 `go2_v2` 本身不是一个足够稳的基线：在 result/log 中多次出现异常重开、训练不稳定、迁移后策略容易被打散等现象。因此这条路线被弃用。

新的路线改为：

```text
直接训练平地多速度策略
-> 修复多速度静止下压/抖动问题
-> 使用多速度策略作为斜坡课程初始化
-> 选取早期稳定 checkpoint 作为斜坡 demo
```

这条路线比从 `go2_v2` 迁移更稳定，也更适合展示“平地多速度能力 + 地形课程微调”的完整流程。

## 多速度策略

### 训练目标

多速度策略覆盖：

- 前进/后退
- 侧移
- 转向
- 接近零命令时静止

### 主要问题

多速度训练早期出现两个现象：

1. 静止时轻微抖动。
2. 零命令或近零命令时身体下压。

后续判断：下压并不完全是失稳，更像策略在多速度任务中学到的“准备行动/降低重心”姿态。但旧配置里的 `base_height_target=0.3` 会明显加重下压，因为 Go2 正常站立高度更接近 `0.34m`。

### 关键修改

多速度训练中将高度目标从：

```python
"base_height_target": 0.3
```

改为：

```python
"base_height_target": 0.34
```

并在静止奖励中尝试加入“低于目标高度才惩罚”的 `height_drop` 项，进一步减少下压。最终观察是：轻微下压仍然存在，但属于可接受的准备姿态，不再继续强行压制。

### 推荐多速度模型

```text
logs/go2-multivel-height034-stillheight-v1/model_4999.pt
```

这个模型作为后续斜坡课程初始化效果最好。

## 斜坡课程

### 路线选择

斜坡课程没有先做环境大重构，而是优先产出 demo：

```text
使用多速度 checkpoint 初始化 Go2EnvDR 斜坡课程
固定正向速度命令
从低坡度课程开始
通过短训 + 评估选择最好的中间 checkpoint
```

没有直接把斜坡命令改成完整多速度范围，因为当前 `Go2EnvDR` 的 `forward_progress`、`uphill_progress`、`completion`、`stall` 都默认沿 track 正向推进。直接加入后退、侧移或大转向命令会让奖励目标冲突。

### 训练尝试

#### A1 smoke

```text
exp_name: go2-slope-from-multivel-height034-a-smoke
ckpt: logs/go2-multivel-height034-stillheight-v1/model_4999.pt
slope_levels: 0,3,5,8,10
target_lin_vel: 0.35
```

结果：能学到 `forward_progress/uphill_progress`，但后半段 episode length、tracking 和 orientation 下降，策略变散。

#### A2 stable

```text
exp_name: go2-slope-from-multivel-height034-a2-stable
ckpt: logs/go2-multivel-height034-stillheight-v1/model_4999.pt
slope_levels: 0,3,5,8
target_lin_vel: 0.30
reset_std: 0.15
finetune_lr: 1e-4
```

这是当前最佳路线。训练后期仍然会退化，但中间 checkpoint 表现很好。

#### A3 low entropy

```text
exp_name: go2-slope-from-multivel-height034-a3-lowentropy
entropy_coef: 0.001
```

结果：成功压住探索扩散，但整体表现不如 A2。说明问题不是单纯 entropy，而是课程/奖励在后期会把策略带偏。

## 结论

当前项目的可发布结果是：

1. 平地多速度 Go2 策略。
2. 基于多速度策略初始化的 8deg 斜坡通过式 demo。
3. 斜坡评估中 0/3/5/8deg 全部成功，8deg 可录像展示。

暂时不切换 SAC。PPO 已经能训出可用策略，当前瓶颈主要是奖励和课程设计，而不是 PPO 完全不适配。

后续如果继续提升斜坡上限，优先做：

```text
forward_progress: 2.5 -> 1.5
uphill_progress: 2.0 -> 1.0
completion: 2.0 -> 1.0
randomize_on_max_success: False -> True
```

目标是降低后期训练把 gait 稳定性换成短期 progress 的倾向。
