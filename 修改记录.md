## 修改斜坡课程：选择方案 A，用多速度 height034 模型做低坡度 smoke test

### 对应训练名称

```text
go2-slope-from-multivel-height034-a-smoke
```

### 背景问题

当前已有多速度平地模型：

```text
logs/go2-multivel-height034-stillheight-v1/model_4999.pt
```

该模型具备比较稳定的平地全向 gait，并且已经把 Go2 站立高度目标从 `0.3` 修正到 `0.34`。现在希望用它作为斜坡/台阶等课程训练起点，尽快产出地形 demo。

风险在于：多速度训练使用 `Go2EnvMultiVel`，斜坡/台阶课程使用 `Go2EnvDR`。两者虽然 actor 观测结构同为 45D、动作维度同为 12D，checkpoint 技术上可以加载，但任务分布、奖励函数和 reset/curriculum 逻辑并不完全一致。

### 三个方案分析

方案 A：最快斜坡 demo 路线，最终选择。

```text
ckpt: logs/go2-multivel-height034-stillheight-v1/model_4999.pt
slope_levels: 0,3,5,8,10
target_lin_vel: 0.35
command: 固定正向前进，不引入侧移/转向/后退
base_height_target: 0.34
```

优点是改动小、奖励逻辑一致、最适合快速验证多速度 gait 能否迁移到斜坡。缺点是它不是完整“多速度斜坡”，而是用多速度平地策略作为斜坡初始化。

方案 B：保留少量多速度范围的斜坡训练，暂缓。

```text
vx: 0.25~0.45
vy: -0.10~0.10
wz: -0.25~0.25
zero_cmd_prob: 0.03~0.05
```

这个方案更接近“多速度斜坡”，但 `Go2EnvDR` 的 `forward_progress`、`uphill_progress`、`completion`、`stall` 都默认沿 track 正向推进。如果直接加入负向、侧向或较大转向命令，可能导致速度 tracking 和课程推进奖励互相打架，降低 smoke test 判读效率。因此不作为第一步。

方案 C：斜坡主线 + 低台阶并行探索，暂缓。

台阶可以从 `0.03,0.05,0.08m` 的低高度开始尝试，但当前台阶链路缺少 height scan 和 foot clearance reward，直接做正式台阶 demo 风险高。它可以作为低成本探索，但不应该抢斜坡主线的时间。

### 为什么选择 A

当前目标是尽快产出斜坡 demo，而不是先做环境架构统一。统一 `Go2EnvMultiVel` 和 `Go2EnvDR` 从工程上更干净，但会消耗时间，且短期不直接产出 demo。方案 A 保留了多速度模型的 gait 初始化优势，同时避免改动 `Go2EnvDR` 的课程推进假设，是当前最实际的路线。

结合官方资料重新判断：

1. RSL-RL 官方配置说明中，`entropy_coef` 是 entropy regularization，`GaussianDistribution` 的 `learn_std=True` 默认会让 std 可学习。因此调低 entropy 可以压探索，但不会修复奖励目标冲突。
2. legged_gym 的 terrain curriculum 是“走够距离升难度、走不够降难度”的 game-inspired curriculum；当前项目也是类似逻辑，但 `randomize_on_max_success=False` 会让成功后长期停留在最高难度附近，可能加速后期退化。
3. legged_gym rough terrain 默认使用 `measure_heights=True` 和前方 height measurement 网格；当前项目 45D 观测没有地形高度扫描，对斜坡还可以试，对台阶会更吃力。
4. legged_gym 的典型奖励里有 `feet_air_time`、`collision`、`action_rate`、tracking、姿态等组合，而当前斜坡任务额外加入较强 `forward_progress=2.5`、`uphill_progress=2.0`、`completion=2.0`。这可能导致 policy 优先“推进/过线”，但不一定保持原来的稳定 gait。

## 修改多速度训练：静止阶段增加低高度惩罚，继续压低下压幅度

### 对应训练名称

```text
go2-multivel-height034-stillheight-v1
```

### 问题现象

`go2-multivel-height034-v1` 已经把 `base_height_target` 从 `0.3` 改到 `0.34`，训练结果显示下压确实减少，但没有完全消失。说明高度目标修正有效，但多速度策略仍会在零速度或近零速度命令下选择轻微降低重心。

这个现象更像是 policy 在多速度任务里学到的“准备行动”姿态：为了随时响应前进、后退、侧移和转向，静止时不追求完全直立，而是保留一个更低、更稳、更方便起步的姿态。

### 为什么继续修改

上一版只改全局 `base_height_target`，但 `_reward_stillness()` 本身仍没有直接约束“静止时不要低于目标高度”。它惩罚的是：

```python
vel_norm
ang_norm
joint_motion
action_bias
dof_deviation
```

这些项能减少静止抖动和动作偏移，但不能明确区分“站在目标高度”和“轻微蹲低但更稳”。如果 policy 发现轻微下压可以减少晃动或提升命令切换稳定性，它仍可能保留下压。

这次修改选择在 stillness 内增加非对称高度惩罚：只惩罚低于 `base_height_target` 的部分，不惩罚高于目标的部分。这样可以专门压制静止下蹲，同时避免全局提高 `base_height` 权重导致运动 gait 被过度干扰。

### 修改内容

文件：`go2_env_multivel.py`

在 `_reward_stillness()` 中新增：

```python
height_drop = torch.clamp(
    self.reward_cfg["base_height_target"] - self.base_pos[:, 2],
    min=0.0,
)
```

并加入 stillness 惩罚：

```python
+ height_drop * 2.0
```

完整含义是：只有当命令接近零、且机身高度低于目标高度时，才额外惩罚下压。命令非零时该项不会生效，因为整个 stillness reward 仍乘以：

```python
is_zero_cmd = (cmd_norm < 0.1).float()
```

### 预期判断

这次修改应该主要影响静止段和近零命令段，不应该明显破坏前进、后退、侧移、转向 gait。如果训练后出现以下结果，说明方向正确：

1. 静止时 base 高度更接近 `0.34m`。
2. 下压幅度继续减少。
3. `rew_stillness` 负值可能略有增加，但不应成为主导损失。
4. `episode_length` 仍接近满长。
5. `tracking_lin_vel` 和 `tracking_ang_vel` 不明显下降。

如果下压减少但起步变慢或 tracking 明显变差，说明 `height_drop * 2.0` 可能偏强，可以降到 `1.0` 或 `1.5`。

### 训练后观察

本次训练到 `model_4999.pt` 后，静止时仍然有微微下压，并且仍带有“准备行动”的姿态特征。相比前一版，继续强行压制下压的收益已经不明显；如果再提高静止高度约束，可能会牺牲多速度命令切换时的起步自然性和 gait 稳定性。

当前结论：暂时接受这个轻微下压，不继续加大静止高度惩罚。后续如果要把这个模型作为斜坡或台阶课程起点，优先关注它的多速度 gait 能否迁移到地形任务，而不是继续围绕静止姿态调参。

## 修改多速度训练：对齐 Go2 站立高度目标，减少静止下压

### 对应训练名称

```text
go2-multivel-height034-v1
```

### 问题现象

上一版多速度静止策略已经能保持较好的存活和速度跟踪，但在零速度命令或接近零速度命令时，身体仍会出现下压。这个下压不是立即摔倒，也不是明显的失控，而是 policy 在静止段主动把机身压低到一个更低、更稳的姿态。

这次训练后观察到：下压仍然存在，但幅度相比 `base_height_target=0.3` 的版本确实减少，说明高度目标对这个现象有直接影响。

### 原因判断

旧配置里：

```python
"base_height_target": 0.3
```

而 Go2 默认站立时的合理 base 高度更接近 `0.34m` 左右。虽然 `base_init_pos` 是 `[0.0, 0.0, 0.42]`，但这是仿真初始化高度，不等价于落地稳定后的机身目标高度。把 reward 目标设为 `0.3` 会让 policy 在训练中持续被 `base_height` 奖励牵引到偏低姿态。

这次把目标高度改到 `0.34` 后，下压减少但没有完全消失，说明根因不只是一行高度目标，还和多速度训练目标之间的折中有关：

1. 多速度策略要随时准备响应前进、后退、侧移和转向命令，低一点的重心可能被 policy 学成“准备行动”的起始姿态。
2. `zero_cmd_prob=0.05` 只提供少量显式静止样本，训练主体仍然是运动命令，所以 policy 不会把“完全直立静止”作为唯一最优解。
3. `stillness` 约束惩罚速度、关节运动、动作幅度和默认姿态偏离，但没有单独惩罚“低于目标高度”的静止姿态；只要下压能换来更小晃动或更快起步，policy 仍可能保留一部分下压。
4. `similar_to_default=-0.5` 会拉回默认姿态，但它是全局姿态约束，不是静止阶段的强高度约束，因此在命令切换和准备起步时可能让位给速度跟踪稳定性。

因此当前判断是：`base_height_target=0.3` 是下压偏重的主要原因；改为 `0.34` 后仍存在的轻微下压，更可能是 policy 在多速度任务中学到的“准备行动/降低重心”策略，而不是单纯配置错误。

### 修改内容

文件：`go2_train_multivel.py`

```python
"base_height_target": 0.34
```

同时同步注释：

```python
"base_height": -1.0,   # 保持Go2正常站立高度约0.34m
```

这次只采用方案 1：对齐目标高度。没有加入静止时的额外 `height_drop` 惩罚，也没有加入 action warmup，目的是先验证高度目标本身对下压问题的影响。

### 结论

`go2-multivel-height034-v1` 的结果支持这个判断：高度目标修正有效，但不能完全消除静止下压。后续如果希望进一步压掉静止时的下蹲，优先考虑在 `_reward_stillness()` 中增加“只惩罚低于目标高度”的 `height_drop` 项，而不是继续提高全局 `base_height` 惩罚权重。

当前不建议把轻微下压直接视为失败。如果它满足以下条件，可以暂时接受为多速度策略的准备姿态：

1. 静止时足端没有明显踏步。
2. base 高度均值稳定在 `0.33~0.35m` 附近。
3. 命令切换到运动后能顺滑起步。
4. `episode_length` 和 tracking reward 没有明显下降。

## 修改斜坡训练：记录多速度基线直接迁移失败，并同步 qpos0 警告修复

### 训练结果记录

尝试使用 `go2-multivel-stillness-v2/model_5000.pt` 作为斜坡课程训练起点，最终结果不好：

```text
Learning iteration: 4999/5000
Mean reward: 10.07
Mean episode length: 280.71
Mean action std: 1.02

Episode/rew_tracking_lin_vel: 0.1262
Episode/rew_tracking_ang_vel: 0.0385
Episode/rew_forward_progress: 0.5582
Episode/rew_uphill_progress: 0.1684
Episode/rew_completion: 0.0020
```

同时斜坡训练仍出现：

```text
[Genesis] [WARNING] Neutral robot position (qpos0) exceeds joint limits.
```

### 原因分析

这次失败的核心原因是：`multivel` 和 `slope` 并不是同一个环境类。

```text
多速度训练：Go2EnvMultiVel
斜坡训练：Go2EnvDR
```

虽然两者观测维度都是 45，网络结构可以加载，但任务分布不一致。多速度训练是平地全向速度跟踪，命令覆盖前进、后退、侧移、转向和静止；斜坡训练是单向前进加地形课程，核心奖励来自 `forward_progress`、`uphill_progress`、`completion`、`stall` 等。

从日志看，`action_std = 1.02`，明显高于稳定平地基线的约 `0.31`，说明 fine-tune 后策略被打散；`tracking_lin_vel = 0.1262` 也很低，说明没有学到稳定的斜坡前进跟踪。

### 修改内容

1. 将 `go2_env_dr.py` 的 Go2 URDF 加载方式同步为本地 neutral-safe URDF：

```python
file=ensure_go2_neutral_safe_urdf()
```

2. 在 `scene.build()` 后恢复 calf 真实关节限制：

```python
restore_go2_calf_limits(self.robot, calf_dof_idx)
```

这和多速度环境的 qpos0 warning 修复保持一致：只解决 Genesis build-time 的 neutral qpos 检查，不改变训练时真实 calf 关节范围。

### 结论

`go2-multivel-stillness-v2` 适合作为“平地多速度展示/控制”的基线，但不适合直接作为当前 `Go2EnvDR` 斜坡课程训练的起点。

斜坡训练更建议回到同环境体系内的基线，例如：

```text
logs/go2_v2/model_999.pt
logs/go2-slope-from-go2v2/model_4999.pt
logs/go2-slope-from-go2v2-v2/model_5000.pt
```

后续如果要复用多速度能力，应该先做环境统一：让斜坡训练也基于 `Go2EnvMultiVel` 的观测、命令和 reset 逻辑，再逐步加 terrain curriculum，而不是直接跨环境加载 checkpoint。

## 修改多速度训练：降低静止约束强度，恢复多速度存活稳定性

### 训练结果记录

`go2-multivel-stillness-v2` 最后一轮达到满 episode 存活，说明温和静止约束版本恢复了多速度 gait 的稳定性：

```text
Learning iteration: 5000/5001
Mean reward: 35.37
Mean episode length: 1001.00
Mean action std: 0.31

Episode/rew_tracking_lin_vel: 1.8853
Episode/rew_tracking_ang_vel: 0.4738
Episode/rew_lin_vel_z: -0.0166
Episode/rew_ang_vel_xy: -0.0205
Episode/rew_base_height: -0.0011
Episode/rew_action_rate: -0.2035
Episode/rew_similar_to_default: -0.3097
Episode/rew_stillness: -0.0254
```

和上一版 `go2-multivel-stillness-v1` 相比，`v2` 的 `episode_length` 回到 `1001.00`，`action_std` 回落到 `0.31`，tracking reward 也接近旧稳定基线，说明 `zero_cmd_prob=0.05` 与 `stillness=-1.0` 的组合更适合保留多速度能力。

当前仍观察到一个细节：静止时身体会轻微下压。这更像是策略在零速度命令下选择了一个更低、更稳的“准备/防晃”姿态，而不是失稳。原因可能包括：

1. `base_height` 目标是 `0.3`，而初始 `base_init_pos` 是 `0.42`，策略天然会被奖励牵引到更低的身体高度。
2. `stillness` 惩罚了动作幅度、关节速度和默认姿态偏离，但没有单独约束“静止时保持站高”。
3. `similar_to_default` 是全局姿态约束，权重为 `-0.5`，但在零速度段仍可能让位给降低身体高度后的稳定收益。

因此，轻微下压本身不一定是坏事；如果下压后仍能保持足端稳定、无明显踏步、命令切换后能顺滑起步，它可以理解为 policy 学到的静止准备姿态。

### 基线判断

`go2-multivel-stillness-v2` 可以作为后续斜坡或台阶 fine-tune 的候选基线，理由是：

1. 满 episode 存活：`Mean episode length = 1001.00`。
2. 动作随机性合理：`Mean action std = 0.31`，没有出现 v1 后期的动作发散。
3. 速度跟踪强：`tracking_lin_vel = 1.8853`，`tracking_ang_vel = 0.4738`。
4. 静止惩罚已被控制在小幅负值：`rew_stillness = -0.0254`，没有主导训练。

建议作为 slope/stairs 基线时，优先从 `model_5000.pt` 开始 fine-tune，并保持地形训练初期的探索强度偏低，例如 reset std 使用 `0.25~0.35`，避免把已经稳定的平地 gait 打散。

### 背景

`go2-multivel-stillness-v1` 的训练后期表现不理想。对比旧实验 `go2-multivel_morestable2` 后，主要差异如下：

```text
go2-multivel-stillness-v1  episode_length tail ≈ 536
go2-multivel_morestable2   episode_length tail ≈ 1000

go2-multivel-stillness-v1  mean_reward ≈ 7
go2-multivel_morestable2   mean_reward ≈ 36

go2-multivel-stillness-v1  action_std ≈ 0.72
go2-multivel_morestable2   action_std ≈ 0.30
```

`stillness-v1` 在 3000 iter 左右曾经接近满存活，但随后 episode length 明显回落，同时 `action_std` 升高，说明策略后期变得更不稳定。

### 原因分析

上一版为了让机器狗静止时不原地踏步，新增了 20% 显式全零命令：

```python
zero_cmd_mask = torch.rand(self.num_envs, ...) < 0.2
new_cmd[zero_cmd_mask, :] = 0.0
```

这个比例对多速度训练来说太高。它显著改变了训练分布，使策略花过多采样预算学习“站立模式”，挤占了前进、后退、侧移、转向这些 gait 的学习。

同时上一版 stillness 内部惩罚较强：

```python
joint_motion * 0.08
action_bias * 0.3
dof_deviation * 0.1
```

再叠加 `stillness: -2.0`，会让静止目标过强。策略一边要在零命令时站死，一边要在非零命令时快速跟踪全向速度，二者在命令切换阶段产生冲突，最终表现为动作标准差升高、tracking reward 下降、存活时间变短。

### 修改内容

1. 将显式零命令采样比例从 `20%` 降到 `5%`：

```python
"zero_cmd_prob": 0.05
```

2. 将 `stillness` reward scale 从 `-2.0` 降到 `-1.0`：

```python
"stillness": -1.0
```

3. 将 `_reward_stillness()` 内部惩罚改为温和版本：

```python
joint_motion * 0.04
action_bias * 0.15
dof_deviation * 0.03
```

4. `go2_env_multivel.py` 不再写死零命令比例，而是从 `command_cfg["zero_cmd_prob"]` 读取，后续调参只需要改训练配置。

### 设计原因

这次修改的目标不是完全取消静止学习，而是把静止样本作为少量但明确的训练信号。这样 policy 仍然会见到全零命令，并受到 stillness 惩罚约束，但不会让静止目标主导整个多速度训练。

温和版参数优先保护多速度 gait 的存活稳定性，再逐步改善静止抖动。后续如果静止仍然不够稳，可以优先小幅提高 `zero_cmd_prob` 到 `0.08` 或把 `stillness` 调到 `-1.2`，不建议直接回到 `0.2 / -2.0`。

### 影响范围

本次改动只影响多速度训练：

```text

go2_train_multivel.py
```

斜坡、台阶、通用训练入口未改动。

## 修改多速度训练：修复 Genesis qpos0 关节限位警告

### 背景

训练启动时 Genesis 会输出：

```text
[Genesis] [WARNING] Neutral robot position (qpos0) exceeds joint limits.
```

排查后发现，多速度训练配置里的默认站姿没有越界。当前默认值中 calf 为 `-1.5 rad`，Genesis 内置 Go2 URDF 的 calf 合法范围是 `[-2.7227, -0.83776]`，该值在范围内。

外部资料也能对上这个范围：Unitree 官方 Go2 页面给出的运动范围是 body `-48~48°`、thigh `-200~90°`、shank `-156~-48°`；第三方 Go2 文档也列出了 calf `-156~-48°`。换算到弧度后，calf 上限约为 `-0.83776 rad`，因此 `0.0 rad` 确实不是合法膝关节姿态。

真正触发 warning 的位置是 Genesis `scene.build()` 阶段。Genesis 解析 URDF 时会先使用 revolute 关节默认 neutral qpos `0.0` 做检查，而 Go2 的 calf/knee 关节上限是 `-0.83776`，因此 `0.0` 不在合法范围内，于是产生 warning。

### 修改内容

1. 新增 `go2_urdf_utils.py`。
2. 首次运行时从 Genesis 内置 Go2 资源复制一份本地 URDF 到：

```text
assets/go2_neutral_safe/urdf/go2.urdf
```

3. 本地 URDF 只在 build-time 层面把 `*_calf_joint` 的上限临时放宽到 `0.0`，用于通过 Genesis 的 neutral qpos 检查。
4. `scene.build()` 之后立即调用 `restore_go2_calf_limits()`，把仿真运行时 calf 真实关节范围恢复为：

```text
[-2.7227, -0.83776]
```

5. 多速度环境 `go2_env_multivel.py` 改为加载本地 neutral-safe URDF，而不是直接加载 Genesis 包里的 Go2 URDF。

### 设计原因

这次没有修改 conda 环境中的 Genesis 安装文件，避免污染第三方包。也没有永久放宽训练时的 calf 关节限制，因为这会改变机器人真实可达空间，影响训练物理约束。

采用“build 前放宽、build 后恢复”的方式，只解决 Genesis neutral qpos warning，不改变训练过程中的真实关节限制。

### 影响范围

本次改动只影响多速度训练环境：

```text
go2_env_multivel.py
go2_train_multivel.py
```

斜坡、台阶、通用训练入口未改动。

启动多速度训练：

```bash

```

观察启动阶段是否还出现：

```text

```
